# 04 — 接棒指南（接棒 AI 必读）

> 你是接棒开发这个项目的 AI。这个系统管理的是用户的真金白银。
> 本文的规则优先级高于你的任何默认行为和"更好的想法"。

## 铁律（违反任何一条 = 停止工作并报告用户）

1. **绝不写直接调用交易所下单 API 的代码。** 所有下单经 freqtrade。哪怕"只是一个小脚本测试一下"也不行。
2. **绝不让 LLM 的输出成为开仓信号。** LLM 只能：产出研报、产出复盘、投否决票（VETO）。见 ADR-002。
3. **绝不修改资金升级状态机的门槛**（ADR-005 的天数/比例/回撤条件）。用户本人要求改也要先复述风险再执行。
4. **绝不在配置中关闭** `stoploss_on_exchange`、删除任何 Protection、或把 `max_open_trades` 调到 >3。
5. **绝不把 API key 写进代码或 git。** 发现历史提交里有密钥 → 立即报告，指导用户轮换。
6. **交易所 API key 必须是无提现权限的。** 部署文档和 RUNBOOK 里反复强调这一点。
7. **回测结果好 ≠ 可以实盘。** 任何策略变更都要重新走状态机，从 dry-run 开始。
8. **不确定就问，不要猜。** 涉及钱的歧义，猜错的代价是不对称的。

## 工作方式

- **任务来源唯一**：`03-tasks.md`。按 Phase 和编号顺序做，完成后勾选并简要记录（改动了哪些文件、验收怎么过的）。
- **决策查询顺序**：`03-tasks.md`（做什么）→ `02-design.md`（怎么做）→ `01-architecture.md`（为什么）→ `00-research-summary.md`（依据数据）。四层都没答案 → 问用户。
- **标 🧠 的任务**：这些是策略逻辑、风控路径等高风险任务。如果你做了两次尝试仍不满足 DoD，建议用户切回强模型处理，不要硬凑一个"看起来能跑"的版本。
- **每个任务的 DoD 是硬性的**：测试没过、覆盖率不够、回测没含 2022 熊市——都算没完成。不要标记完成。
- **遵循用户全局规则**：TDD（先写测试）、覆盖率 ≥80%、不可变数据模式、文件 <800 行、函数 <50 行、无魔法数字。

## 常见坑（前任 AI 替你踩过的）

- **freqtrade 的 dry-run 和 backtest 结果会有差异**是正常的（撮合模型不同），偏差需要解释而不是消除。
- **未来函数（lookahead bias）**：任何用到 `shift(-1)`、当根 K 线 high/low 参与信号计算的写法都可疑。每个策略必须跑 `freqtrade lookahead-analysis`。
- **hyperopt 过拟合**：只在训练段跑 hyperopt（P1.4 的划分），留出段只许看一次。反复用留出段调参 = 留出段失效。
- **LLM JSON 输出不稳定**：所有 LLM 输出必须过 Pydantic 校验，失败要有重试与降级，绝不 `json.loads` 裸解析后直接用。
- **AI 服务的可用性语义**：AI 服务挂掉时策略必须默认放行（fail-open）并告警。AI 是保险丝不是闸门——反过来会导致 AI 故障阻塞整个交易系统。
- **网络**：本机环境遇到 GitHub/API 超时，先 `export https_proxy=http://127.0.0.1:7890 http_proxy=http://127.0.0.1:7890 all_proxy=socks5://127.0.0.1:7890`。
- **时区**：所有时间戳统一 UTC 存储，展示层再转。加密市场没有"收盘价"，日线边界用 UTC 00:00。

## 项目背景速览

用户：稳健型个人投资者，目标年化 8-15%、最大回撤 <20%，看重纪律自动化甚于策略聪明度。曾主观炒美股亏钱，因此本系统的存在意义就是**用流程防止人性犯错**——你写的每一行风控代码都在保护用户不被自己伤害。认真对待。
